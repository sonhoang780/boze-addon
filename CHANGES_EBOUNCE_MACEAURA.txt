 1. MaceAura.java — "Cách 4": two-tick fold → strike (Y/XZ position spoof)
================================================================================

VẤN ĐỀ GỐC (verify từ MaceItem.canSmashAttack, decompiled 26.1.2):
    canSmashAttack = (attacker.fallDistance > 1.5) && !attacker.isFallFlying()
  Cả 2 điều kiện phải đúng TRÊN SERVER tại đúng tick xử lý packet attack.

VÌ SAO CÁC CÁCH TRƯỚC THẤT BẠI:
  • fallDistance được cập nhật ĐỒNG BỘ ngay trong packet handling
    (ServerGamePacketListenerImpl.handleMovePlayer -> doCheckFallDamage ->
     Entity.checkFallDamage: `if (ya < 0) fallDistance -= ya`).
  • NHƯNG isFallFlying (shared flag 7) chỉ đổi trong aiStep -> updateFallFlying,
    chạy SAU toàn bộ packet batch mỗi server tick.
  • => Gửi StatusOnly(onGround=true) + attack trong CÙNG 1 tick: aiStep chưa chạy
    nên isFallFlying vẫn = true lúc attack -> canSmashAttack=false VÀ 6b6t tịch thu
    mace ("văng mace", không base damage). Đây là bug của Cách 1/2/3.

CÁCH 4 (HIỆN TẠI) — tách làm 2 server tick:
  FOLD tick : gửi StatusOnly(onGround=true) ở EventTick.Post (sau vanilla move packet,
              nên onGround=true là giá trị server giữ cho tick đó).
              -> cuối tick aiStep: updateFallFlying thấy onGround=true ->
                 canGlide()=false -> setSharedFlag(7,false). isFallFlying đã FALSE
                 server-side, và KHÔNG tự bật lại (chỉ START_FALL_FLYING mới bật).
  Đợi FoldDelay tick (>=2) : chống packet bunching khi ping cao, đảm bảo server
              chạy aiStep giữa FOLD và STRIKE.
  STRIKE tick: gửi PosRot hạ Y xuống (threshold+0.2), onGround=false ->
              doCheckFallDamage cộng đồng bộ vào fallDistance; isFallFlying vẫn false
              -> attack -> canSmashAttack TRUE -> SMASH, mace không bị văng.
              Rồi gửi START_FALL_FLYING để resync server với client (vốn chưa hề
              ngừng glide) -> giữ nguyên độ cao, không fall damage.

  Client KHÔNG hề di chuyển (chỉ gửi fake packet) -> ElytraFly giữ độ cao.
  Y được miễn trong check "moved wrongly" (yDist luôn bị zero ở SGPLI) -> Y-spoof
  không bao giờ rubber-band. XZSpoof fake X/Z = tọa độ target -> reach gate
  (isWithinEntityInteractionRange) luôn pass. Đây cũng là cơ chế player đặt web
  từ trên cao.

STATE MACHINE: IDLE -> APPROACH -> FOLD -> STRIKE -> IDLE

OPTIONS (default):
  Range 4.0, ApproachRange 24, VerticalRange 20, MinHeight 2.5,
  SmashThreshold 1.7 (server min 1.5; cao hơn = damage nhiều hơn nhưng reach giảm),
  FoldDelay 2 (TĂNG lên 3-4 nếu ping >200ms — quan trọng nhất khi test),
  Delay 250ms, AutoTarget on, SilentSwap on, AttributeSwap off,
  XZSpoof on (fake X/Z để reach luôn pass),
  ElytraFly on, FlySpeed 1.0, VertSpeed 0.8.

CÁCH TEST:
  - Mace trong hotbar, elytra đang mặc/deploy.
  - Bay lên trên target >= MinHeight (2.5) block, để module tự fold->strike.
  - ĐÚNG khi hoạt động: thấy vòng xoáy gió + mảnh block quanh target, target mất
    máu, và mình KHÔNG rớt xuống.
  - Nếu mace bị văng / không damage -> FoldDelay đang quá thấp so với ping -> tăng.
  - Test phải trên server thật (Dummy là client-side, server không xử lý attack).

 2. EBounce.java — elytra bhop, dynamic pitch + ceiling controller
================================================================================

PHYSICS THẬT (updateFallFlyingMovement, decompiled LivingEntity 26.1.2):
    liftForce = cos^2(pitch)
    vy += 0.08 * (-1 + 0.75*cos^2θ)                 // gravity/lift
    if (vy < 0): convert = -0.1 * vy * cos^2θ        // CHỈ khi đang rơi (vy<0)
                 horizontal += convert (theo hướng nhìn);  vy += convert
    horizontal *= 0.99 ;  vy *= 0.98                 // drag
  => gain ngang mỗi dive ~= 0.1 * cos^2θ * Δh  (Δh = độ cao rơi).
     Conversion TẮT khi đang bay lên (vy>=0).

ĐÍNH CHÍNH: lời khuyên "pitch 35°" trước đây SAI (mô hình thiếu drag+feedback).
  Không có 1 góc đúng duy nhất — là trade-off cos^2θ (phẳng, gain/block cao) vs
  thời gian rơi (drag). Optimum thực nghiệm ~25-45° tùy độ sâu dive.

CÁC THAY ĐỔI ĐÃ LÀM:
  (a) RisePitch: 80° -> 5° (mặc định). Lúc bay lên conversion bị tắt nên nhìn xuống
      vô ích, chỉ làm gravity mạnh giết độ cao bounce. Nhìn phẳng = bounce cao hơn
      = dive sâu hơn = nhanh hơn (khi trời mở). Trong hầm thì CeilingGuard lo phần
      không đụng trần, nên vẫn để phẳng được.

  (b) DynamicDive (toggle, default on): pitch biến thiên theo |vy| trong lúc dive.
      Dốc lúc apex (DiveApexPitch 65°) để dồn |vy| nhanh, rồi flatten về
      DiveFastPitch (14°) khi |vy| lớn -> cú convert cuối đạt cos^2θ≈1 = burst lớn.
      Interp: t = min(1, |vy|/DiveVyFull[1.1]); pitch = lerp(apex, fast, t).
      LƯU Ý: chỉ hiệu quả khi dive SÂU (|vy| kịp lên cao). Hầm thấp -> |vy| không
      kịp build -> DynamicDive gần như vô tác dụng (đã xác nhận với user).

  (c) GlidePitch (30°): pitch dive cố định, chỉ dùng khi DynamicDive OFF.

  (d) CeilingGuard (toggle, default on) + CeilingMargin (1.4): controller bám trần.
      Mỗi tick raycast thẳng lên (ceilingClearance, scan 12 block). Trong computePitch:
        bay lên + còn trống trên đầu        -> giữ risePitch (leo)
        bay lên + trần <= CeilingMargin     -> CẮT NGAY sang dive pitch (không đụng trần)
        đang rơi                            -> dive pitch
      => tự lấp đầy đúng khoảng trống của hầm cho dù trần cao bao nhiêu, không cần
      nhập số. Nếu vẫn cụng trần (quán tính vy>0) -> tăng CeilingMargin (2.0-2.5).

STATE MACHINE: IDLE -> JUMPING -> TRIGGERING -> GLIDING (-> JUMPING khi chạm đất)
  Pitch override: set ở EventTick.Pre (player.tick() gửi packet với pitch này),
  khôi phục camera thật ở EventTick.Post -> không lộ ra hình ảnh.

KỲ VỌNG TỐC ĐỘ (đã tính từ physics):
  - Terminal (glide liên tục, trời mở vô hạn độ cao): ~60-74 b/s — không thực tế.
  - Ground bounce trời mở, tune tốt: ~33-38 b/s. 40 b/s là rìa trên, không ổn định.
  - Trong hầm có trần: bị Δh khóa cứng -> tốc độ trần bị trần đè. Hầm càng thấp
    càng chậm. 40 b/s trong hầm bedrock thấp gần như BẤT KHẢ bằng bounce thuần
    -> cần firework boost.

KHUYẾN NGHỊ THEO ĐỘ CAO HẦM:
  - Hầm thấp (trần <= ~3 block): TẮT DynamicDive, GlidePitch phẳng ~15-20°
    (dive nông -> tối đa cos^2θ = nhìn phẳng). Bật CeilingGuard.
  - Hầm cao / trời mở: BẬT DynamicDive (mới phát huy). RisePitch phẳng.

HƯỚNG CÒN BỎ NGỎ (chưa làm, chờ user quyết sau khi test):
  - Nếu cần 40+ ổn định trong hầm cao: "air-oscillation" (không chạm đất, climb<->dive
    quanh 1 dải độ cao) thay vì bounce floor.
  - Nếu hầm quá thấp: auto-firework boost.
